# Contributing to SafeSphere

Thank you for your interest in contributing to SafeSphere! This document provides guidelines for contributing to the project.

## 🚀 Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/YOUR_USERNAME/safespera.git`
3. Create a new branch: `git checkout -b feature/your-feature-name`
4. Make your changes
5. Test your changes thoroughly
6. Commit your changes: `git commit -m "Add: your feature description"`
7. Push to your fork: `git push origin feature/your-feature-name`
8. Create a Pull Request

## 📝 Commit Message Guidelines

Use clear and descriptive commit messages:

- `Add: new feature or functionality`
- `Fix: bug fix`
- `Update: changes to existing features`
- `Refactor: code refactoring`
- `Docs: documentation updates`
- `Style: formatting, missing semicolons, etc.`
- `Test: adding or updating tests`

## 🎯 Development Guidelines

### Code Style
- Follow PEP 8 for Python code
- Use meaningful variable and function names
- Add comments for complex logic
- Keep functions small and focused

### Testing
- Test your changes on multiple devices
- Ensure responsive design works (mobile, tablet, desktop)
- Test on different browsers (Chrome, Firefox, Safari, Edge)
- Verify all existing features still work

### Documentation
- Update README.md if adding new features
- Add docstrings to new functions
- Update setup guides if changing configuration

## 🐛 Reporting Bugs

When reporting bugs, please include:
- Description of the bug
- Steps to reproduce
- Expected behavior
- Actual behavior
- Screenshots (if applicable)
- Browser/device information
- Error messages or logs

## 💡 Suggesting Features

We welcome feature suggestions! Please:
- Check if the feature already exists
- Describe the feature clearly
- Explain why it would be useful
- Provide examples if possible

## 🔍 Code Review Process

All contributions go through code review:
1. Automated checks must pass
2. Code must follow project guidelines
3. Changes must be tested
4. Documentation must be updated
5. At least one maintainer approval required

## 📋 Pull Request Checklist

Before submitting a PR, ensure:
- [ ] Code follows project style guidelines
- [ ] All tests pass
- [ ] New features have tests
- [ ] Documentation is updated
- [ ] Commit messages are clear
- [ ] No merge conflicts
- [ ] Changes are tested on multiple devices

## 🎨 UI/UX Guidelines

When contributing to the frontend:
- Maintain consistent design language
- Ensure accessibility (WCAG 2.1 AA)
- Test on mobile devices
- Use existing color palette and spacing
- Add smooth transitions and animations
- Optimize images and assets

## 🔒 Security

- Never commit sensitive data (.env files, API keys, passwords)
- Report security vulnerabilities privately
- Follow Django security best practices
- Validate all user inputs
- Use CSRF protection

## 📞 Questions?

If you have questions:
- Check existing documentation
- Search closed issues
- Open a new issue with the "question" label

## 🙏 Thank You!

Your contributions help make disaster preparedness education accessible to everyone!
